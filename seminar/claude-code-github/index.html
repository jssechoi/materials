01 · 왜 해야 하는가

달라진 건 하나 — 이제 도구에게 "말로" 시킵니다

실리콘밸리에서 시작된 흐름의 핵심은 "AI가 코드를 대신 짠다"가 아니라, 사람이 자연어로 목적을 말하면 AI가 실제 작업을 수행한다는 것입니다.

예전 방식

  • 내가 직접 문서를 열고, 찾고, 정리하고, 옮겨 적는다
  • 반복 작업을 매번 수작업으로
  • "할 줄 아는 사람"에게 부탁해야만 가능

지금 방식

  • 목적을 말로 전달하면 AI가 파일을 읽고·쓰고·정리
  • 반복 작업을 지시 한 번으로 자동화
  • 코드를 몰라도 개발자용 도구의 생산성을 그대로
01 · 무엇이 좋은가

비개발자에게 특히 강력한 네 가지

자동화

보고서 초안, 문서 취합, 데이터 정리 같은 반복 업무를 지시 한 번으로 끝냅니다.

일괄 처리

여러 문서·표·이미지를 한꺼번에 읽어 요약·변환·작성합니다.

팀 공유

결과물을 GitHub에 남겨 팀 누구나 최신본을 보고 이어받습니다.

재현·추적

무엇을·언제·왜 바꿨는지 기록으로 남아 되돌리고 검토할 수 있습니다.

핵심은 "AI가 똑똑해서"가 아니라 목적 → 재료 → 기록 → 검토의 흐름을 만들어 두면 같은 품질을 반복해서 얻는다는 점입니다. 오늘 그 흐름을 만듭니다.

02 · 환경 설정 — 2

터미널을 열고 Claude Code를 설치합니다

터미널(명령 입력창) 열기

irm https://claude.ai/install.ps1 | iex

관리자 권한 불필요 · 자동 업데이트 · 가장 간단

대안

# Node.js가 있다면
npm install -g @anthropic-ai/claude-code

winget install Anthropic.ClaudeCode
02 · 환경 설정 — 3

실행하고 로그인합니다

# 작업할 폴더에서 실행
claude
  • 처음 실행하면 로그인 안내가 나옵니다 → 세션 안에서 /login 입력 시 브라우저로 인증
  • 인증이 끝나면 대화창에 한국어로 지시를 입력하면 됩니다
  • 모델을 바꾸려면 /model opus 처럼 입력

비개발자를 위한 마음가짐

  • 명령은 슬래시(/)로 시작하고, 나머지는 평범한 한국어 문장이면 됩니다
  • 완벽한 문장이 아니어도 됩니다 — 목적과 재료만 분명하면 AI가 되묻습니다
  • 겁내지 마세요. 실수해도 Esc로 멈추고 다시 시키면 됩니다
02 · 환경 설정 — 4

꼭 알아 둘 명령어 8가지

명령어하는 일터미널(CLI) 대안
/loginAnthropic 계정으로 로그인첫 실행 시 자동 안내
/model사용할 AI 모델 변경 (opus·sonnet·haiku)claude --model opus
/effort사고 깊이 조절 (low·medium·high·max)claude --effort high
/compact길어진 대화를 요약해 맥락을 정리세션 안에서만
/clear대화를 비우고 새 대화 시작세션 안에서만
/statusline하단 상태줄 꾸미기 (모델·경로·git 표시)세션 안에서만 (설정 파일 가능)
/usage사용량·요금(플랜 한도) 확인세션 안에서만
/resume이전 대화를 골라 이어가기claude --resume · -r

입력창에 / 만 치면 전체 명령 목록이 자동완성으로 뜹니다.

02 · 환경 설정 — 5

손이 빨라지는 단축키

  • Esc — 진행 중인 응답 즉시 중단
  • Esc Esc — 이전 메시지로 되감기·수정(rewind)"아, 잘못 말했다" 할 때 앞 메시지를 고쳐 다시 보냄
  • Shift + Tab — 모드 순환(기본 → 자동수락 → 계획)
  • ↑ / ↓ — 이전에 입력한 내 메시지 다시 불러오기
  • @ 파일명 — 특정 파일을 대화에 첨부 (예: @report.md)
  • ! 명령 — 뒤의 셸 명령을 바로 실행 (예: !dir)
  • # 내용 — 뒤 내용을 기억(메모)으로 저장
  • Ctrl + V — 복사한 스크린샷을 그대로 붙여넣기

가장 자주 쓰는 건 Esc(멈춤), EscEsc(되감기), Shift+Tab(모드 전환) 셋입니다. 이것만 익혀도 충분합니다.

02 · 환경 설정 — 6 · AUTO MODE

자동 모드 — 매번 묻지 않고 알아서 실행

파일을 만들고 고치는 작업을 매번 확인받지 않고 AI가 이어서 진행하는 모드입니다. (정식 명칭 accept edits / 자동 수락)

켜는 법

Shift+Tab 한 번 → 하단에 "accept edits on" 표시.
시작부터 켜려면 claude --permission-mode acceptEdits

언제 쓰나

방향이 분명하고 반복적인 작업, 결과를 나중에 한꺼번에 검토할 때. 속도가 크게 빨라집니다.

주의

파일이 자동으로 바뀝니다. 중요한 폴더는 GitHub로 백업해 두면 언제든 되돌릴 수 있습니다.

비개발자 권장 순서: 먼저 계획 모드로 계획을 확인하고 → 자동 모드로 실행. 위험한 셸 명령은 자동 모드에서도 대개 한 번 더 확인을 받습니다.

02 · 환경 설정 — 7 · PLAN MODE

계획 모드 — 먼저 계획만, 승인 후 실행

AI가 파일을 건드리지 않고 읽고 조사만 해서 "이렇게 하겠습니다"라는 계획을 먼저 보여 줍니다. 당신이 승인해야 실제 작업이 시작됩니다.

쓰는 법

  • 켜기: Shift+Tab 두 번 (기본→자동수락→계획)
  • 또는 시작 시 claude --permission-mode plan
  • 계획 제시 후 승인 / 수정 요청 / 계속 계획 중 선택

왜 좋은가 (특히 비개발자)

  • 어려운 코드 대신 글로 된 계획을 검토하므로 이해하기 쉬움
  • 큰 작업 전에 방향을 먼저 맞춰 실수를 예방
  • "잠깐, 그건 아니야"를 실행 전에 말할 수 있음

이 세미나 자료도 계획 모드로 계획을 먼저 승인받은 뒤 만들었습니다. (→ 09장에서 그 과정을 공개합니다)

04 · GitHub 연동 — 1

GitHub는 "기록이 남는 팀 공유 드라이브"입니다

코드·문서를 담는 클라우드 저장소(repository)입니다. 익숙한 것에 비유하면 —

공유 드라이브

파일을 한곳에 모아 팀 전체가 같은 최신본을 봅니다.

+ 타임머신

누가·언제·무엇을 바꿨는지 전부 기록되어 과거로 되돌릴 수 있습니다.

+ 결재·리뷰

반영 전에 확인·의견(PR)을 거쳐 안전하게 합칩니다.

우리에게 GitHub가 특별한 이유: Claude(AI)가 읽고 쓰는 작업 공간이 되기 때문입니다. 결과물을 여기에 남기면 다음 작업 때 AI가 그대로 이어받습니다.

04 · GitHub 연동 — 2

무엇을 위해 쓰는가

한 곳에서 협업

흩어진 파일·버전 혼란 없이, 팀이 하나의 저장소에서 같은 자료로 일합니다.

이력 추적·되돌리기

변경마다 스냅샷이 남아 "3일 전 상태"로 복구하거나 비교할 수 있습니다.

검토(PR) 후 반영

바로 덮어쓰지 않고 확인을 거쳐 합쳐 실수를 줄입니다.

AI의 지식 저장소

산출물·규칙·기록을 저장소에 두면 AI가 맥락을 이해하고 이어서 작업합니다.

04 · GitHub 연동 — 4

commit · push · pull · PR — 4단어의 원리

내 PC로컬 — commit → 스냅샷 기록commit — push → GitHub원격 ← pull — 최신본 받기pull
commit바뀐 내용을 메시지와 함께 스냅샷으로 기록= 문서 "저장"
push내 PC의 커밋들을 GitHub로 올리기= 공유 드라이브에 업로드
pullGitHub의 최신 내용을 내 PC로 내려받기= 최신본 다운로드
PR"이 변경을 반영해도 될까요?" 검토 요청 → 승인 후 병합= 반영 전 결재

실무에선 "작업 내용을 커밋하고 푸시해줘", "이 변경으로 PR 만들어줘"처럼 말하면 AI가 처리합니다.

05 · 문서 관리·질의 — 1

저장소에 문서를 두고 관리합니다

코드뿐 아니라 기획·회의록·스펙·가이드를 저장소 안 docs/ 폴더에 두면, 문서도 이력·검토·공유의 대상이 됩니다.

이렇게 둡니다

저장소/
├─ docs/ # 최종 문서
│ ├─ 기획안.md
│ └─ 회의록_260715.md
├─ scratch/ # 중간 메모
└─ CLAUDE.md # AI 규칙서

그러면 이렇게 씁니다

  • "이 저장소의 PVI API 스펙을 요약해줘"
  • "회의록들에서 내게 배정된 할 일만 뽑아줘"
  • "기획안과 회의록이 서로 안 맞는 부분 있어?"
05 · 문서 관리·질의 — 2

원리 — AI는 "필요한 부분만 찾아 읽습니다"

Claude는 저장소를 통째로 외우지 않습니다. 질문을 받으면 그때그때 검색 → 파일 찾기 → 해당 부분 읽기로 관련된 곳만 꺼내 근거로 삼습니다.
질문무엇을 묻는가→ 검색관련 단어 찾기→ 읽기해당 파일만→ 근거 있는 답출처와 함께

그래서 폴더·파일 이름이 명확할수록 더 빠르고 정확합니다. "그 문서"보다 "docs/기획안.md의 3장"처럼 짚어 주면 최고입니다.

06 · 메모(.md) 기록 — 1

AI는 생각·기록을 .md 파일로 남기고 서로 링크합니다

.md(마크다운)는 서식이 있는 일반 텍스트입니다. AI는 중간 결과와 메모를 md로 적고, 문서끼리 [[다른문서]]로 연결해 다음에 그 링크만 따라가 빠르게 이해합니다.

사람도 읽기 쉬움

특수 프로그램 없이 어디서나 열리는 글자 그대로의 문서입니다.

변경 비교가 됨

GitHub가 어디가 바뀌었는지 줄 단위로 보여 줍니다(diff).

사람·AI 공용어

사람이 검토하기도, AI가 이해하기도 가장 편한 형식입니다.

왜 중요한가: 결과물을 md로 정리해 두면, 그것이 곧 다음 작업의 재료이자 AI의 기억이 됩니다. 작업이 쌓일수록 저장소가 똑똑해집니다.

06 · 메모(.md) 기록 — 2

어떻게 관리하나

자리를 나눕니다

  • docs/ — 완성된 최종 문서
  • scratch/ — 중간 계산·초안 등 임시 기록
  • 한 파일 = 한 주제, 이름에 날짜·주제를 담기

습관을 정합니다

  • 살아 있는 문서는 링크로 연결, 낡은 스크래치는 정리·삭제
  • 결과를 낼 때 "먼저 md로 정리하고" → 그다음 최종 형태로
  • "이번 작업 요약을 docs/에 md로 남겨줘"라고 지시

규칙이 거창할 필요 없습니다. "최종은 docs, 임시는 scratch, 하나에 한 주제" 세 가지면 충분합니다.

07 · CLAUDE.md — 원리

CLAUDE.md는 AI에게 주는 "항상 지킬 규칙서"

저장소에 CLAUDE.md 파일을 두면, Claude가 대화를 시작할 때마다 자동으로 읽어 그 내용을 규칙으로 따릅니다. 매번 설명하지 않아도 됩니다.
  • 자동 로드 — 세션마다 먼저 읽힙니다. 별도 명령 불필요
  • 계층 구조 — 전역 규칙 + 프로젝트 규칙이 합쳐짐
  • 충돌 시 — 더 가까운 프로젝트 규칙이 우선

07 · CLAUDE.md — 활용

무엇을 적고, 어떻게 만드나

이런 것을 적습니다

  • 사용 언어 ("문서·커밋은 한국어로")
  • 폴더 구조·자주 쓰는 경로
  • 지켜야 할 규칙·금지사항
  • 회사 환경 (예: "프록시 경유", "gh CLI 사용")

이렇게 만듭니다

  • 프로젝트 폴더에서 /init → AI가 초안 생성
  • 초안을 읽고 내 규칙을 덧붙여 다듬기
  • 새 규칙이 생기면 # 로 그때그때 추가

실제 예 — 이 워크스페이스의 CLAUDE.md에는 "한국어 프로젝트는 한국어로 작성", "GitHub는 gh CLI 사용", "폴더별 역할"이 적혀 있어, AI가 매번 같은 방식으로 일합니다.

08 · 명령 예시 — 1 · AUTO MODE

예시 ① 로컬 문서 + GitHub를 함께 이해시켜 작성

> 내 로컬 PC의 문서 C:\내문서\report_template.xlsx 를 보고,
  현재 연결된 github AXLAB/c4-content-intelligence-agent 의 내용을
  이해해서 작성해줘. 오류가 있으면 알려줘. (auto mode)

AI가 하는 일

  • 로컬 엑셀 양식을 읽고 구조 파악
  • 연결된 저장소를 검색·이해
  • 자동 수락으로 결과물 작성
  • 불일치·오류를 함께 보고

왜 auto mode

방향이 분명하고("양식대로 채워") 결과를 나중에 한 번에 검토하면 되는 작업이라, 매 편집을 확인받지 않고 빠르게 끝내기 위함입니다.

08 · 명령 예시 — 2 · PLAN MODE

예시 ② 스크린샷을 검토해 "계획부터" 세우게

> 이번 주 작업한 내용을 docx 문서로 작성하고 싶어.
  첨부한 스크린샷을 검토해서 작성한 뒤에, 어떤 내용을 쓸지
  계획을 세워서 나한테 맞는지 질문해. (plan mode)

AI가 하는 일

  • 스크린샷을 분석해 작업 내용 파악
  • 문서 구성(목차) 계획을 제시
  • "이 방향 맞나요?" 확인 질문
  • 승인 후 docx로 작성

왜 plan mode

결과 형태와 방향이 중요할 때, 바로 만들지 않고 계획을 먼저 맞춰 헛수고를 막습니다. 원하는 문서가 한 번에 나옵니다.

08 · 실전 팁 · 스크린샷

스크린샷을 AI에게 건네는 법

Win + Shift + S — 영역 캡처
화면 일부를 드래그로 선택 → 클립보드에 복사됩니다 (가장 많이 씀)
Win + PrtSc — 전체 화면 저장
전체 화면을 사진 폴더에 파일로 저장합니다
Claude에 붙여넣기
입력창에서 Ctrl + V → 캡처한 이미지가 그대로 첨부됩니다. 파일을 끌어다 놓기도 가능

화면·오류 메시지·디자인 시안을 말로 설명하기 어려울 땐 캡처 → 붙여넣기가 가장 빠릅니다. "이 화면 보고 표로 정리해줘"처럼요.

09 · 이 문서의 제작 — 1

이 자료를 만들 때 실제로 준 "지시문"

특별한 기술이 아니라, 목적·대상·구성·재료를 한 번에 적어 전달했을 뿐입니다.
# 목적·대상·스타일
"HTML로 세미나 자료를 만들려고 해. 스타일은 첨부한 PDF와 비슷하게.
 목표는 Claude Code와 GitHub를 연동해 업무를 자동화하는 방법.
 대상은 비개발자야.

# 구성 (11개 챕터)
 1. 왜 해야 하는가 … 2. 설치·명령어·모드 … 4. GitHub 연동 …
 7. CLAUDE.md 활용 … 8. 명령 예시 … 11. 마무리
 챕터별 길이는 알아서 구성해줘."

참고: 첨부한 PDF는 회사 보안(DRM)으로 잠겨 있어 AI가 열지 못했습니다. 그래서 스타일은 사내 기존 슬라이드 덱 + 삼성 브랜드 폰트로 대체해 만들기로 함께 결정했습니다.

09 · 이 문서의 제작 — 2

AI가 밟은 실제 작업 흐름

1
지시 접수 · 참고 PDF 열기 시도
→ DRM 잠금 확인 후 사용자에게 즉시 보고
2
워크스페이스 탐색
쓸 수 있는 삼성 폰트와 기존 슬라이드 덱 2종을 발견 (스타일 근거 확보)
3
확인 질문 3개
스타일·형식·폰트를 물어 사용자가 결정 (슬라이드 덱 / 기존 덱 기반 / 삼성 폰트)
4
계획 수립 → 승인 plan mode
11개 챕터의 슬라이드 구성을 계획으로 제시하고 승인받음
5
HTML 덱 작성 · 검토
기존 덱의 엔진·컴포넌트를 재사용해 자체완결 HTML 생성 → 브라우저로 확인

지금 보고 계신 이 슬라이드가 바로 그 결과물입니다.

10 · 작업 흐름 요약
